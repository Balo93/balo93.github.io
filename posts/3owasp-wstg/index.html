OWASP WSTG: La guía definitiva para auditorías Web profesionales

Publicado 05/06/2026 Actualizado 05/06/2026
Por Alvaro Gavilanes
4 min de lectura

Imagina que estás auditando una tienda en línea. La aplicación tiene registro de usuarios, buscador de productos, carrito de compras, pasarela de pagos y rastreo de envíos. ¿Por dónde empiezas? ¿Pruebas el formulario de login? ¿La barra de búsqueda? ¿La API de pagos?

Con docenas de superficies de ataque en una sola aplicación web, es increíblemente fácil cometer dos errores: probar lo mismo dos veces o, peor aún, olvidar clases enteras de vulnerabilidades.

Para resolver este caos, el Open Web Application Security Project creó la Web Security Testing Guide (WSTG). Mientras que la mayoría conoce a OWASP por su famoso Top 10, la WSTG va muchísimo más profundo: es un framework comunitario que organiza las pruebas web en más de 90 casos de prueba concretos divididos en 12 categorías.

Las 12 Categorías del WSTG

Estas doce categorías cubren absolutamente toda la superficie de ataque de una aplicación moderna. Cada caso de prueba está numerado (por ejemplo, WSTG-INPV-01 para Reflected Cross-Site Scripting) e incluye instrucciones paso a paso de qué probar y cómo probarlo:

  1. Information Gathering (Recopilación de información)
  2. Configuration and Deployment Management (Gestión de configuración y despliegue)
  3. Identity Management (Gestión de identidades)
  4. Authentication (Autenticación)
  5. Authorization (Autorización)
  6. Session Management (Gestión de sesiones)
  7. Input Validation (Validación de entradas)
  8. Error Handling (Manejo de errores)
  9. Cryptography (Criptografía)
  10. Business Logic (Lógica de negocio)
  11. Client-Side Testing (Pruebas del lado del cliente)
  12. API Testing (Pruebas de APIs)

Enfoque basado en riesgo: La WSTG no es una lista plana de fallos; prioriza las vulnerabilidades según su explotabilidad e impacto, ayudando a los pentesters a enfocar sus esfuerzos donde realmente importa.

Seguridad en todo el Ciclo de Vida del Software (SDLC)

A diferencia de otros frameworks de pentesting que se aplican únicamente cuando el sistema ya está terminado, la WSTG destaca porque integra la seguridad a lo largo de las 5 fases del desarrollo de software (SDLC).

Veamos cómo se aplica en una empresa real que está construyendo un portal web:

[Antes del Desarrollo] ➔ [Diseño] ➔ [Desarrollo] ➔ [Despliegue] ➔ [Mantenimiento]

Fase 1: Antes de que comience el desarrollo

Se establecen los requisitos de seguridad y obligaciones regulatorias desde el primer día.

  • Ejemplo: Definir que el portal debe cumplir con PCI DSS (por procesar pagos) y fijar tiempos máximos aceptables para la aplicación de parches de seguridad.

Fase 2: Durante la definición y el diseño

Se revisa la arquitectura antes de escribir una sola línea de código mediante modelos de amenazas.

  • Ejemplo: Identificar que la API de pagos será un objetivo crítico y diseñar desde el inicio controles de rate-limiting (limitación de peticiones) y validación estricta de entradas.

Fase 3: Durante el desarrollo

El código se examina activamente mediante revisiones y walkthroughs utilizando las guías de OWASP.

  • Ejemplo: Los desarrolladores revisan el módulo de autenticación contra los casos de prueba WSTG-ATHN y descubren a tiempo un fallo crítico: los tokens de restablecimiento de contraseña no expiran.

Fase 4: Durante el despliegue

Se verifican los controles de seguridad directamente en el entorno de producción o staging mediante un pentesting tradicional.

  • Ejemplo: El equipo de seguridad ejecuta pruebas para comprobar que se cambiaron las credenciales por defecto, que TLS está bien configurado y que no hay endpoints de depuración (debug) expuestos.

Fase 5: Durante el mantenimiento y operaciones

La seguridad se mantiene post-lanzamiento con revisiones periódicas, especialmente tras cada actualización.

  • Ejemplo: Si tres meses después se añade una nueva función de “recomendación de productos”, se vuelven a ejecutar los casos de prueba de la WSTG específicos para garantizar que esa actualización no introdujo nuevos fallos.

Pros y Contras de OWASP WSTG

🟢 Lo mejor

  • Exhaustivo y práctico: Más de 90 casos de prueba con procedimientos específicos y resultados esperados. Es un mapa real, no principios abstractos.
  • Siempre actualizado: Al ser mantenido por una comunidad global, se adapta constantemente a nuevas arquitecturas como aplicaciones de una sola página (SPAs) y microservicios.

🔴 Lo difícil

  • Consume muchos recursos: Ejecutar los 90+ casos de prueba en su totalidad puede ser inviable para equipos pequeños o con presupuestos limitados.
  • Requiere alta especialización: Algunos test exigen conocimientos muy avanzados en áreas como análisis criptográfico o fallos complejos en la lógica de negocio.
  • El peligro de la “Mentalidad de Checklist”: Existe el riesgo de que el tester se vuelva mecánico, marcando casillas en automático sin detenerse a pensar críticamente en el riesgo global de la aplicación.

Conclusión

La WSTG es la base perfecta para cualquier auditoría web profesional. Te da la estructura necesaria para no dejar cabos sueltos, pero recuerda: las mejores auditorías ocurren cuando usas la WSTG como cimiento y aplicas el pensamiento crítico más allá de la lista de control.
